Meridian

Binary stock outcome markets on Solana devnet.

Trade tokens that pay $1.00 USDC if a MAG7 stock closes at or above a strike today, and $0.00 if it closes below. Non-custodial. In-program order book. Pyth-settled at 16:00 ET.

Components

Five pieces. Each does one thing.

Anchor program

Single Solana program. Owns every piece of on-chain state. Instructions: initialize_config, create_strike_market, mint_pair, place_order, cancel_order, settle_market_manual, admin_settle, redeem, pause, unpause.

Pros
  • One source of truth for state
  • Atomic Buy No path is feasible
  • Property tests for invariants
Cons
  • Custom matcher is less battle-tested
  • Stack pressure from large accounts
  • Compute budget per tx is finite

In-program order book

Zero-copy slabs. Bids descending, asks ascending. FIFO at price. Depth 64 per side. USDC and Yes escrow ATAs owned by book PDA. Matching via cranker.

Pros
  • Atomic mint+sell in one ix
  • No off-chain listing coordination
  • Fits Solana CPI create limit
Cons
  • Less depth than mature DEXs
  • We own matching-engine bugs
  • Requires zero-copy discipline

Frontend (Next.js)

App Router on Render. Pages: landing, markets grid, trade page with order book, portfolio, history. TanStack Query for chain reads, Zustand for UI, wallet adapter for Phantom and Solflare (Wallet Standard auto-discovery). Branded UsdcBase bigint type for money.

Pros
  • SSR landing, client trading
  • Type-safe Anchor client from IDL
  • Sub-2s order book polling
Cons
  • Buffer/Node polyfill weight
  • Wallet adapter UX is opinionated

Automation service

Node 20 + croner on Render. Two crons in America/New_York. Morning: Pyth read + create_strike_market per strike. Settlement: 15-min retry window + admin_settle per market. NYSE calendar hard-coded. JSON logs.

Pros
  • Sub-100ms cron drift
  • Idempotent on every job
  • Slack alert on give-up
Cons
  • Single-instance (no HA in v1)
  • Keypair custody required

Pyth oracle

Pull model. Off-chain: Hermes for previous close and settlement price. On-chain (slice 2): pyth-solana-receiver-sdk for staleness + confidence-band verify inside settle_market.

Storage walkthrough

Meridian has one order book per market. It trades YES tokens priced in USDC. The "NO view" in the user interface is a pure transform of the same book with prices mirrored across $1. NO trades happen in a single signed transaction via the atomic buy_no and sell_no instructions, which bundle a pair mint or burn with a fill against the book.

Each step below shows the five places state lives: three user keypairs (John, Kerry, Bob), each represented by the bundle of token accounts their keypair owns, plus two program-owned PDAs (the market PDA's USDC vault, and the order book PDA with its two escrow token accounts). A glowing border means the bucket changed this step. Inline arrows like $0.60 ← $1.00 show the previous value.

On-chain reality · wallet vs ATA

A "wallet" (Phantom, Solflare) is just a keypair. The keypair owns exactly one System Account, which holds the SOL balance and pays rent and fees. Fungible token balances never live in the System Account.

Each SPL mint a user holds requires its own Token Account, whose owner field points at the keypair's pubkey. The conventional, deterministically-derived address for the pair (keypair, mint) is the Associated Token Account (ATA). Solflare aggregates all of a keypair's ATAs into one UI screen, so the "USDC: $0.60" row inside Kerry's card on these diagrams is actually Kerry's USDC ATA, not "USDC inside her wallet".

An ATA must exist before tokens can land in it. The first time anyone sends YES to Kerry, the transaction includes a createAssociatedTokenAccountIdempotent instruction. Rent for a 165-byte Token Account is about 0.00203 SOL, one time, refundable on close.

Order book vocabulary

Bid = a parked order to buy at a maximum acceptable price. "Kerry's bid at 40¢" means Kerry will pay at most 40¢ per YES.

Ask (also called offer) = a parked order to sell at a minimum acceptable price. "John's ask at 40¢" means John will sell for at least 40¢ per YES.

Cross = when an incoming bid lands at or above the best resting ask (or an incoming ask lands at or below the best resting bid), the matching engine fills immediately. By price-time priority, the fill price is the resting order's price, not the incoming order's.

So in step 2 below, John's ask at 40¢ meets Kerry's resting bid at 40¢ and the trade clears at 40¢ on the spot.

Step 5 · Resolution (two endings)

Same starting state as the end of step 4: two pairs outstanding, the market PDA's USDC vault ATA holding $2. The oracle (Pyth) posts an outcome. Each winning holder calls redeem(side, qty): the program burns N tokens of the winning mint from the holder's winning-side ATA, then transfers N USDC from the vault ATA to the holder's USDC ATA. Losing-side tokens remain in their ATAs as worthless dust; they can be closed for the rent refund but the mint authority prevents any further redemption.

How the price gets set

Three layers, simplest to deepest. All three are simultaneously true in Meridian's single-book design.

Mechanical

Price = the last fill on the order book. With no recent fill, mid price = (best bid + best ask) / 2. Orders match by price-time priority: better price first, earlier order first at the same price.

Probabilistic

A YES token pays $1 if YES wins and $0 otherwise, so its expected value is the probability of winning. The book's YES price in dollars equals the implied probability. $0.60 means the market thinks YES has about 60% odds; the same order in the NO panel renders as $0.40.

Duality by construction

YES + NO = $1 is enforced at the program level, not by external arbitrage. Every pair mints from exactly $1 and redeems for exactly $1 from the collateral vault, and there is no second book where prices could diverge. The NO view is the same book, transformed.

All six effects land in one signed transaction. If the best resting bid is gone or below min_bid_price_ticks, the whole transaction reverts and no token account changes. Note that the YES transfer goes directly from Bob's freshly-minted YES ATA to Kerry's YES ATA — the book's YES escrow ATA is not touched because Kerry's resting order was a bid, not an ask.

Trade-offs

Each one is a "we accept X, knowing Y, mitigated by Z, triggers a revisit when W".

In-program book vs Phoenix CPI

Why we accept it: atomic Buy No needs single-program revert; Phoenix listing required off-chain coordination. When it would bite: obscure slab-walk bug nobody happens to write a test for. Mitigation: property tests on conservation (USDC + Yes) across 1000 random sequences planned in slice 3.5.

Pyth pull vs Switchboard

Why we accept it: first-class MAG7 equity feeds with confidence intervals; pull model lets us post fresh price at settle. When it would bite: sustained Hermes outage > 15 min during US close. Mitigation: admin override after 1-hour delay with the override flag recorded on chain.

Position constraint as UX rule

Why we accept it: market makers calling mint_pair transiently hold both Yes and No; blocking at program level would break legit liquidity. When it would bite: user uses CLI directly and confuses themselves. Mitigation: frontend disables Buy Yes when holding No; portfolio labels pending sells.

Devnet-only for v1

Why we accept it: PRD requires devnet for the pass; constitution forbids mainnet for v1. When it would bite: devnet liquidity is whatever test wallets put in. Mitigation: mainnet path is a documented bonus, separate review.
