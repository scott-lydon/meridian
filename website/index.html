Meridian

Binary stock outcome markets on Solana devnet.

Trade tokens that pay $1.00 USDC if a MAG7 stock closes at or above a strike today, and $0.00 if it closes below. Non-custodial. In-program order book. Pyth-settled at 16:00 ET.

Components

Five pieces. Each does one thing.

Anchor program

Single Solana program. Owns every piece of on-chain state. Instructions: initialize_config, create_strike_market, mint_pair, place_order, cancel_order, settle_market_manual, admin_settle, redeem, pause, unpause.

Pros
  • One source of truth for state
  • Atomic Buy No path is feasible
  • Property tests for invariants
Cons
  • Custom matcher is less battle-tested
  • Stack pressure from large accounts
  • Compute budget per tx is finite

In-program order book

Zero-copy slabs. Bids descending, asks ascending. FIFO at price. Depth 64 per side. USDC and Yes escrow ATAs owned by book PDA. Matching via cranker.

Pros
  • Atomic mint+sell in one ix
  • No off-chain listing coordination
  • Fits Solana CPI create limit
Cons
  • Less depth than mature DEXs
  • We own matching-engine bugs
  • Requires zero-copy discipline

Frontend (Next.js)

App Router on Vercel. Pages: landing, markets grid, trade page with order book, portfolio, history. TanStack Query for chain reads, Zustand for UI, wallet adapter for Phantom + Solflare. Branded UsdcBase bigint type for money.

Pros
  • SSR landing, client trading
  • Type-safe Anchor client from IDL
  • Sub-2s order book polling
Cons
  • Buffer/Node polyfill weight
  • Wallet adapter UX is opinionated

Automation service

Node 20 + croner on Render. Two crons in America/New_York. Morning: Pyth read + create_strike_market per strike. Settlement: 15-min retry window + admin_settle per market. NYSE calendar hard-coded. JSON logs.

Pros
  • Sub-100ms cron drift
  • Idempotent on every job
  • Slack alert on give-up
Cons
  • Single-instance (no HA in v1)
  • Keypair custody required

Pyth oracle

Pull model. Off-chain: Hermes for previous close and settlement price. On-chain (slice 2): pyth-solana-receiver-sdk for staleness + confidence-band verify inside settle_market.

Trade-offs

Each one is a "we accept X, knowing Y, mitigated by Z, triggers a revisit when W".

In-program book vs Phoenix CPI

Why we accept it: atomic Buy No needs single-program revert; Phoenix listing required off-chain coordination. When it would bite: obscure slab-walk bug nobody happens to write a test for. Mitigation: property tests on conservation (USDC + Yes) across 1000 random sequences planned in slice 3.5.

Pyth pull vs Switchboard

Why we accept it: first-class MAG7 equity feeds with confidence intervals; pull model lets us post fresh price at settle. When it would bite: sustained Hermes outage > 15 min during US close. Mitigation: admin override after 1-hour delay with the override flag recorded on chain.

Position constraint as UX rule

Why we accept it: market makers calling mint_pair transiently hold both Yes and No; blocking at program level would break legit liquidity. When it would bite: user uses CLI directly and confuses themselves. Mitigation: frontend disables Buy Yes when holding No; portfolio labels pending sells.

Devnet-only for v1

Why we accept it: PRD requires devnet for the pass; constitution forbids mainnet for v1. When it would bite: devnet liquidity is whatever test wallets put in. Mitigation: mainnet path is a documented bonus, separate review.
